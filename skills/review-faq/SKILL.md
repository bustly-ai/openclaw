# Review FAQ

The best FAQs are not written out of thin air — they grow from real reviews and genuine questions.

## Problem Solved

Common issues with product page FAQs:
- Questions are not authentic and fail to address users' hesitation points;
- Answers sound too official, unlike what a real person would ask;
- The same concern keeps appearing in the review section, yet no one organizes it onto the page.

The goal of this skill is:
**Extract FAQs from reviews, negative feedback, Q&A, and common concerns, and output answer drafts that sound more like real user language.**

## When to Use

- The request is fundamentally about reviews, refunds, retention, subscriptions, or public reply workflows, and a decision depends on ranking evidence, diagnosing gaps, or prioritizing fixes.
- The team could take action immediately if FAQ Generator From Reviews returns a clear verdict, deliverable set, and next-step list.
- The user may describe the business problem without naming the skill; route semantically rather than by exact skill name.

## Input Requirements

- User reviews / negative feedback / Q&A
- Product facts and feature boundaries
- Common concerns (sizing, effectiveness, target audience, shipping, returns, etc.)
- Optional: prohibited statements or claim boundaries

## Workflow

1. Define the decision question, review window, and the one KPI or business signal that matters most.
2. Normalize the evidence into a compact table so facts, assumptions, and missing fields are separated.
3. Rank the gaps or options by business impact, reversibility, and confidence.
4. Convert the diagnosis into P0 / P1 / P2 actions tied to evidence, not generic best practices.
5. End with a validation loop so the operator knows exactly how to recheck if the fix worked.
6. Separate product issue, expectation issue, policy issue, and service issue before recommending a response.

## Quality Standards

- Questions should sound like what a real user would ask.
- Answers should be clear, concise, and credible.
- Do not fabricate efficacy claims without supporting evidence.
- Output should be ready to go directly into the PDP or help center.

Most users with sensitive skin reported it is gentle and non-irritating. We recommend patch-testing behind the ear first. [To be added: link to third-party test report]. Suggested placement: PDP detail page collapsible section + customer service quick replies.

## Missing Data Protocol

- Ask for the smallest missing set that would change the recommendation materially.
- If the core blocker is missing (sample size, timeframe, or source of feedback), stop short of a hard verdict and ask for it explicitly.
- If only secondary inputs are missing, continue in checklist or hypothesis mode and label every assumption.
- End low-confidence outputs with the single most valuable next data request instead of hiding uncertainty.

- Separate observed facts, user-provided facts, and assumptions.

- Prefer the shortest useful answer shape that an operator can actually hand off.
- Do not over-generalize from one screenshot, one SKU, one creator, or one time window.
- Keep the top 3-5 actions only; backlog items belong in a secondary list.
- Preserve useful positive signals, not only complaints or failure themes.
