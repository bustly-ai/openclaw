# Storefront Design

## Overview

Go beyond loose design suggestions. Return a page structure, component guidance, breakpoint logic, proof hierarchy, CTA rules, and accessibility checks so a builder or designer can move directly into execution.

## When to Use

- A storefront or landing page needs structure, section hierarchy, or component guidance before build.
- The team needs a page spec that balances conversion, readability, and responsive behavior.

## Do Not Use When

- Do not turn this into a broad brand strategy or copywriting exercise.
- Do not ignore mobile, accessibility, or build constraints.
- Do not prescribe aesthetic direction without tying it back to conversion or clarity goals.

## Required Inputs

- Page type and primary conversion goal
- Traffic source, audience, and trust blockers
- Build constraints, CMS / theme limits, and device mix
- Existing proof assets, offer blocks, and no-go patterns

## Workflow

1. Define the conversion goal and page role in the journey.
2. Set the section hierarchy: hero, proof, education, social proof, offer, objections, CTA, and footer logic.
3. Specify component rules: typography hierarchy, spacing, button treatment, proof modules, and breakpoint behavior.
4. Add accessibility and QA checks: tap targets, contrast, heading logic, and mobile scannability.
5. End with a handoff list for builder or design execution.

## Deliverables

- Page structure and component spec
- Responsive and breakpoint guidance
- Accessibility and QA checklist
- Build handoff priorities

## Missing Data Protocol

- If the page role is unclear, ask whether the page is acquisition, product detail, or launch support first.
- If build constraints are unknown, provide a theme-friendly default and flag the assumption.

## Validation Loop

- Check that each section earns its place in the page flow.
- Verify mobile behavior, accessibility, and CTA clarity before finalizing the spec.
